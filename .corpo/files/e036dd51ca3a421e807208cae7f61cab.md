## Overview
<!-- thread:1b64cb9a -->

The corpo GUI should be deployable as a static site on GitHub Pages, hosted directly from any corpo repo. Users who haven't installed the CLI can still read and comment on corpo files from a browser. The GitHub Contents API replaces the local server as the data layer.

## Goals

1. Corpo GUI is hosted and accessible in-browser for users without the CLI
2. Users can engage according to config-defined read/comment/write permissions
3. Supports private and enterprise (GHES) repositories

## Non-Goals

- File creation, deletion, or navigation reordering (write-heavy operations stay CLI-only for now)
- Real-time collaboration / conflict resolution UI

## Contracts
<!-- thread:8b3c1924 -->

- Any corpo repo gets a hosted GUI at `{owner}.github.io/{repo}/` with no extra configuration
- Feature parity with the local GUI for all supported operations (reading, threading)
- No backend infrastructure required
- Public repos ship first as an intermediate QA milestone; private and GHES support follows

# User Flows

## Corpo Initialization

Author runs `corpo init` in a new repo. The CLI scaffolds `.corpo/config.json`, the GitHub Actions workflow, and a starter file. On next push to main, the workflow builds and deploys the GUI to GitHub Pages automatically.

## Updating Content

Author edits corpo files locally and pushes to main. The Actions workflow rebuilds and redeploys the GUI. Changes are live within minutes.
<!-- thread:ef54d25d -->

## Viewing Hosted Pages

Visitor navigates to `{owner}.github.io/{repo}/`. If the repo is public and `permissions.read` is `["*"]`, no login is required. The GUI renders navigation and file content fetched from the GitHub Contents API.
<!-- thread:ca49c4ba -->

## Sharing Hosted Pages

Author copies any GUI URL (e.g. `{owner}.github.io/{repo}/#/files/{id}`) and shares it. The recipient opens the link directly ‚Äî hash routing ensures the page loads without a server-side redirect.

## Commenting on Hosted Pages

Visitor clicks to add a thread. If not logged in, the app initiates the Device Flow ‚Äî visitor authorizes in a new tab, then returns. Once authenticated and permitted per `permissions.comment`, the comment is written back to the file via the GitHub Contents API as a commit.
<!-- thread:16712c84 -->

# Technical Design

## Data Layer

This feature motivates a **unified adapter interface** for all corpo CRUD operations ‚Äî a shared TypeScript interface (e.g. `CorpoAdapter`) implemented by both a `LocalAdapter` (wrapping the local server) and a `GitHubAdapter` (wrapping the GitHub Contents API). The GUI switches adapters at runtime based on context; the CLI already uses an analogous pattern. Defining this interface as a shared type across packages enforces feature parity and prevents the two surfaces from drifting. Any operation added to one adapter must be added to the other.

Replace `fetch('/api/*')` calls in `api.ts` with GitHub Contents API calls:

| Operation | Endpoint |
|---|---|
| Read navigation | `GET /repos/{owner}/{repo}/contents/.corpo/config.json` |
| Read file | `GET /repos/{owner}/{repo}/contents/.corpo/files/{id}.md` |
| Write file (thread add/resolve) | `PUT /repos/{owner}/{repo}/contents/.corpo/files/{id}.md` |

All responses return base64-encoded content ‚Äî decoded client-side. The file SHA returned on read must be included on write to prevent conflicts. Simultaneous writes get a 409; retry after re-fetching.

The local server mode is preserved. A `VITE_GITHUB_OWNER` / `VITE_GITHUB_REPO` env var pair (set at build time) selects GitHub API mode. If unset, the app falls back to the current local server behavior.

## Auth

Use **GitHub OAuth Device Flow** ‚Äî no backend or redirect URI required:

1. POST to `github.com/login/device/code` with the OAuth App `client_id`
2. Show user the one-time code and `github.com/activate` URL
3. User opens `github.com/activate` in a new tab ‚Äî GitHub prompts them to sign in if not already, then shows a confirmation screen to authorize the app; user enters the 8-character code and clicks Authorize
4. App polls `github.com/login/oauth/access_token` until GitHub confirms authorization
5. Store token in `localStorage`; send as `Authorization: Bearer {token}` on writes

Token scopes: `public_repo` for public repos; `repo` for private repos (grants read access to private content). The OAuth App `client_id` is public ‚Äî safe to embed in the bundle.

**Private repos:** device flow works identically with `repo` scope. The token will authorize API reads against private repos the user has access to.

**Enterprise GitHub (GHES):** device flow is supported but all endpoints are hosted at the enterprise instance ‚Äî replace `github.com` with `{enterprise-hostname}` throughout. The OAuth App must be registered on the GHES instance, not github.com.

Unauthenticated users get read-only access. Write UI elements are hidden or disabled until a token is present.

### Permissions

GitHub login is necessary but not sufficient for access. Permissions are tiered ‚Äî `read`, `comment`, and `write` ‚Äî and configured in `.corpo/config.json` under `permissions`. Each tier takes an array of email patterns. Wildcards are supported: `*` matches anyone, `*@coinbase.com` matches any email from that domain, and exact emails match a single user.

```json
{
  "permissions": {
    "read":    ["*"],
    "comment": ["*@coinbase.com"],
    "write":   ["alice@coinbase.com", "bob@coinbase.com"]
  }
}
```

**Tiers:**
- `read` ‚Äî can view files. Defaults to `["*"]` (public) if absent.
- `comment` ‚Äî can add and reply to threads. Defaults to `["*"]` if absent.
- `write` ‚Äî can edit file content. Defaults to `["*"]` if absent.

Each tier is evaluated independently; a user must satisfy the pattern for a given tier to perform that action. The app fetches the authenticated user's verified email via the GitHub API (`/user/emails`, requires `user:email` scope) and matches it against each tier's patterns client-side.

**Relation to CODEOWNERS:** this permission system is standalone and unrelated to CODEOWNERS. CODEOWNERS governs GitHub's code review request routing ‚Äî it has no concept of read/comment/write on a hosted viewer. The two systems don't overlap.

## Routing

**Invariant: every URL in the app must be directly navigable by pasting it into a browser.** This means the server must return a valid response for any path the app might produce.

GitHub Pages only serves files that exist on disk ‚Äî it has no routing fallback. If a user pastes `owner.github.io/repo/#/files/abc123` into a browser, GitHub Pages serves `index.html` (the root), and the JS reads the hash fragment to render the right file. This works because the hash (`#...`) is never sent to the server ‚Äî the browser handles it entirely client-side.

By contrast, browser history routing (`/files/abc123`) sends the full path to the server. GitHub Pages looks for a file at that path, finds nothing, and returns 404. The app never loads.

**Required change:** switch wouter from browser history mode to hash mode. All internal links update automatically; no server configuration needed.

## Deployment

A GitHub Actions workflow in each corpo repo handles deployment:

```
on: push (main branch)

jobs:
  deploy:
    - bun install
    - vite build  (with VITE_GITHUB_OWNER, VITE_GITHUB_REPO set)
    - deploy dist/ to gh-pages branch via actions/deploy-pages
```

The workflow file is scaffolded by `corpo init` so new repos get it automatically.

## Deployment Requirements

- All static assets (JS, CSS, fonts, images) must load correctly when the app is served at `/{repo-name}/` on GitHub Pages
