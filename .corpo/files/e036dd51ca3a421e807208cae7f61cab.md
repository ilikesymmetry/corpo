# Overview

The corpo GUI should be deployable as a static site on GitHub Pages, hosted directly from any corpo repo. Users who haven't installed the CLI can still read and comment on corpo files from a browser. The GitHub Contents API replaces the local server as the data layer.

## Goals

1. Corpo GUI is hosted and accessible in-browser for users without the CLI
2. Users can engage according to config-defined read/comment/write permissions
3. Supports private and enterprise (GHES) repositories

## Non-Goals

- File creation, deletion, or navigation reordering (write-heavy operations stay CLI-only for now)
- Real-time collaboration / conflict resolution UI
- Accruing tech debt — shortcuts that would require significant rework before launch are not acceptable; the implementation should be production-quality from the start

## Other Requirements

- Any corpo repo gets a hosted GUI at `{owner}.github.io/{repo}/` with no extra configuration
- Feature parity with the local GUI for all supported operations (reading, threading)
- No backend infrastructure required
- Public repos ship first as an intermediate QA milestone; private and GHES support follows

# User Flows

## Corpo Initialization

Author runs `corpo init` in a new repo. The CLI scaffolds `.corpo/config.json`, the GitHub Actions workflow, and a starter file. On next push to main, the workflow builds and deploys the GUI to GitHub Pages automatically. The Pages URL (`{owner}.github.io/{repo}/`) is set as the repository's official Website field on GitHub and added to the top of the README, so visitors can discover it without hunting.

## Updating Content

Author edits corpo files locally and pushes to main. The Actions workflow rebuilds and redeploys the GUI. Changes are typically live within 1–2 minutes. This can be reduced further by caching the bun install step in the workflow.

## Viewing Hosted Pages

Visitor navigates to `{owner}.github.io/{repo}/`. If the repo is public and `permissions.read` is `["*"]`, no login is required. The GUI renders navigation and file content fetched from the GitHub Contents API.

## Sharing Hosted Pages

Author copies any GUI URL (e.g. `{owner}.github.io/{repo}/#/files/{id}`) and shares it. The recipient opens the link directly — hash routing ensures the page loads without a server-side redirect.

## Commenting on Hosted Pages

Authentication is prompted when a visitor:
1. Highlights text (to start a new thread)
2. Replies to an existing thread

If not logged in at that point, the app initiates the Device Flow — visitor authorizes in a new tab, then returns. Once authenticated and permitted per `permissions.comment`, the comment is written back to the file via the GitHub Contents API as a commit.

Resolving a thread requires `permissions.comment`. The resolve action is hidden from users who do not have comment permission.

# Technical Design

## Data Layer
<!-- thread:bae99358 -->

This feature motivates a **unified adapter interface** for all corpo CRUD operations — a shared TypeScript interface (e.g. `CorpoAdapter`) implemented by both a `LocalAdapter` (wrapping the local server) and a `GitHubAdapter` (wrapping the GitHub Contents API). The GUI switches adapters at runtime based on context; the CLI already uses an analogous pattern. Defining this interface as a shared type across packages enforces feature parity and prevents the two surfaces from drifting. Any operation added to one adapter must be added to the other.

Replace `fetch('/api/*')` calls in `api.ts` with GitHub Contents API calls:

| Operation | Endpoint |
|---|---|
| Read navigation | `GET /repos/{owner}/{repo}/contents/.corpo/config.json` |
| Read file | `GET /repos/{owner}/{repo}/contents/.corpo/files/{id}.md` |
| Write file (thread add/resolve) | `PUT /repos/{owner}/{repo}/contents/.corpo/files/{id}.md` |

All responses return base64-encoded content — decoded client-side. The file SHA returned on read must be included on write to prevent conflicts. Simultaneous writes get a 409; conflict resolution is out of scope for V1 — the app should surface the error to the user.

The local server mode is preserved. A `VITE_GITHUB_OWNER` / `VITE_GITHUB_REPO` env var pair (set at build time) selects GitHub API mode. If unset, the app falls back to the current local server behavior.

## Auth

Use **GitHub OAuth Device Flow** — no backend or redirect URI required:

1. POST to `github.com/login/device/code` with the OAuth App `client_id`
2. Show user the one-time code and `github.com/activate` URL
3. User opens `github.com/activate` in a new tab — GitHub prompts them to sign in if not already, then shows a confirmation screen to authorize the app; user enters the 8-character code and clicks Authorize
4. App polls `github.com/login/oauth/access_token` until GitHub confirms authorization
5. Store token in `localStorage`; send as `Authorization: Bearer {token}` on writes

Token scopes: `public_repo` for public repos; `repo` for private repos (grants read access to private content). The OAuth App `client_id` is public — safe to embed in the bundle.

**Private repos:** device flow works identically with `repo` scope. The token will authorize API reads against private repos the user has access to.

**Enterprise GitHub (GHES):** device flow is supported but all endpoints are hosted at the enterprise instance — replace `github.com` with `{enterprise-hostname}` throughout. The OAuth App must be registered on the GHES instance, not github.com.

Unauthenticated users get read-only access. Write UI elements are hidden or disabled until a token is present.

### Permissions

GitHub login is necessary but not sufficient for access. Permissions are tiered — `read`, `comment`, and `write` — and configured in `.corpo/config.json` under `permissions`. Each tier takes an array of email patterns. Wildcards are supported: `*` matches anyone, `*@coinbase.com` matches any email from that domain, and exact emails match a single user.

```json
{
  "permissions": {
    "read":    ["*"],
    "comment": ["*@coinbase.com"],
    "write":   ["alice@coinbase.com", "bob@coinbase.com"]
  }
}
```

**Tiers:**
- `read` — can view files. Defaults to `["*"]` (public) if absent.
- `comment` — can add and reply to threads, and resolve threads. Defaults to `["*"]` if absent.
- `write` — can edit file content. Defaults to `[]` if absent — GitHub repository push access is the effective enforcement gate; corpo's write tier governs whether the GUI surfaces editing UI, not whether the API permits the write.

Each tier is evaluated independently; a user must satisfy the pattern for a given tier to perform that action. The app fetches the authenticated user's verified email via the GitHub API (`/user/emails`, requires `user:email` scope) and matches it against each tier's patterns client-side.

**Relation to CODEOWNERS:** this permission system is standalone and unrelated to CODEOWNERS. CODEOWNERS governs GitHub's code review request routing — it has no concept of read/comment/write on a hosted viewer. The two systems don't overlap.

**Security note:** this permission system is a UI-layer policy, not a security enforcement boundary. All permission checks are evaluated client-side against a config file that is itself publicly readable (or accessible to anyone with repo access). Users can bypass corpo's permission tiers by calling the GitHub Contents API directly. For true access control, rely on GitHub repository visibility and collaborator settings. Corpo's permission tiers govern which actions the GUI surfaces — not what is technically possible.

## Routing
<!-- thread:a9bbb8ed -->

**Invariant: every URL in the app must be directly navigable by pasting it into a browser.** This means the server must return a valid response for any path the app might produce.

GitHub Pages only serves files that exist on disk — it has no routing fallback. If a user pastes `owner.github.io/repo/#/files/abc123` into a browser, GitHub Pages serves `index.html` (the root), and the JS reads the hash fragment to render the right file. This works because the hash (`#...`) is never sent to the server — the browser handles it entirely client-side.

By contrast, browser history routing (`/files/abc123`) sends the full path to the server. GitHub Pages looks for a file at that path, finds nothing, and returns 404. The app never loads.

**Required change:** switch wouter from browser history mode to hash mode. All internal links update automatically; no server configuration needed.

## Deployment

A GitHub Actions workflow scaffolded at `.github/workflows/deploy.yml` by `corpo init` handles deployment. The workflow requires `pages: write` and `id-token: write` permissions for the deploy-pages action to authenticate via OIDC:

```yaml
# .github/workflows/deploy.yml
name: Deploy to GitHub Pages
on:
  push:
    branches: [main]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    steps:
      - uses: actions/checkout@v4
      - uses: oven-sh/setup-bun@v1
      - run: bun install
      - run: bun run build
        env:
          VITE_GITHUB_OWNER: ${{ github.repository_owner }}
          VITE_GITHUB_REPO: ${{ github.event.repository.name }}
      - uses: actions/upload-pages-artifact@v3
        with:
          path: dist/
      - uses: actions/deploy-pages@v4
        id: deployment
```

This pseudocode is a reference implementation — the implementer should exercise judgement to adjust it for changes in Actions runner versions, action versions, or GitHub Pages configuration that may not match this spec exactly. The intent is clear; trust the intent over the literal syntax.

## Deployment Requirements

- All static assets (JS, CSS, fonts, images) must load correctly when the app is served at `/{repo-name}/` on GitHub Pages
- Vite's `base` option must be set to `/{repo-name}/` at build time so asset paths are relative to the subpath, not the root. The implementer should derive this from `VITE_GITHUB_REPO` (e.g. `base: process.env.VITE_GITHUB_REPO ? \`/${process.env.VITE_GITHUB_REPO}/\` : '/'`). This is an engineering implementation detail — the requirement is that assets load correctly; the exact mechanism is left to the implementer's judgement.
