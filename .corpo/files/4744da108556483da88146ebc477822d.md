# Overview

`corpo clone <uri>` brings any external markdown-compatible content into the corpus as a first-class file. The result is a normal corpus file — stable ID, frontmatter, full threading — with the source URI stored in the `remote` field for traceability. `corpo pull <fileId>` re-syncs the body from the source when the original changes. `corpo push <fileId>` sends the local file back to the source for adapters that support writing.
 These three commands replace the `remote_files` config key, which referenced external content live without creating a local copy. Under the new model, all corpus content is local; the clone/pull/push boundary is explicit and always an author action.

## Goals

1. Any markdown-compatible external source can become a corpus file via `corpo clone <uri>`
2. Edited corpus files can be pushed back to their source via `corpo push <fileId>` for adapters that support it
3. `remote_files` is retired and the `~/.corpo` remote content cache is eliminated — both are replaced by local corpus files

## Non-Goals

- Automatic or scheduled sync — pull is always an explicit author action
- Custom source adapters beyond local paths and GitHub files — Google Docs, Confluence, Notion, and similar are out of scope for v1
- Rich-format fidelity — non-markdown source formatting is not preserved; content is treated as plain markdown
- Conflict resolution when the corpus file and source have both changed since the last pull


**V1 scope is two milestones:** same-repo local files (`path:`), then cross-repo GitHub files (`git:`). Custom rich-format sources (Google Docs, Confluence, Notion) are future work.

## Other Requirements

- The source URI is stored in the `remote` frontmatter field so the file can be pulled or pushed without the author tracking the origin separately
- Sync metadata (last sync time, content checksum) is recorded so authors can see how stale their copy is at a glance
- `corpo clone` always creates a new corpus file — if the URI is already tracked by an existing file, the command errors; `corpo pull <fileId>` is the only re-sync path
- All local frontmatter is preserved on pull; only the body is overwritten
- Clone behavior differs by source type:
  - **Non-corpo sources** (local paths): only the body is taken; local frontmatter is entirely the cloning author's own
  - **Corpo sources** (`git:` corpus files): the full raw source file is used as the body — frontmatter delimiters and all — as plain text; it is not parsed as local frontmatter since standard markdown only recognizes the first block; pull simply overwrites the body with the latest raw source content, the same as for non-corpo sources
- Source adapters declare read-only or read-write capability; pushing to a read-only source fails with a clear error
- For cloned corpo sources, push sends the full corpus file (body + frontmatter) back to the source; for non-corpo targets, only the body is sent since the target format has no frontmatter concept
- `remote_files` and the `~/.corpo` remote content cache are both retired — this feature is their complete replacement

# User Flows

## Cloning a Local File

Author runs `corpo clone ./skills/frd/SKILL.md --group "Skills"`. The CLI reads the file, infers the source type from the input (local path → `path:` adapter), and creates a corpus file with the content as body. Title is auto-extracted from the file's first H1 or falls back to the filename; author can override with `--title`. The assigned file ID is printed. The `remote` field and sync metadata are written to frontmatter automatically.

From this point the file is a normal corpus file. Threads can be added, content can be edited. The source path is retained for push.

## Cloning a Remote Corpo File (Cross-Repo Sharing)

Instead of adding an entry to `remote_files`, author runs `corpo clone https://github.com/owner/repo/blob/main/.corpo/files/{id}.md`. The CLI recognizes the GitHub URL, fetches the file via the GitHub Contents API, and creates a local corpus file. The normalized `git:github.com/owner/repo/.corpo/files/{id}.md` URI is stored in `remote`. The local copy can be edited and threaded independently of the source repo.

This replaces the previous cross-repo syncing mental model: rather than a live reference, you clone a snapshot, work with it locally, and optionally push changes back.

## Pushing Changes Back

Author has edited a corpus file originally cloned from a local path. Runs `corpo push {fileId}`. The CLI writes the corpus file body (without frontmatter) back to the path stored in `remote`. For API-based adapters, auth is required.

For a corpus file cloned from a remote corpo file, `corpo push {fileId}` sends the full local corpus file — body and frontmatter — back to the source, so title, description, and thread edits made locally propagate back.

## Pulling Updates

Author knows the source document has been updated since the clone. Runs `corpo pull {fileId}`. The CLI fetches the current source content using the `remote` field, overwrites the body with fresh content, and preserves all frontmatter. Thread anchors referencing shifted line numbers become dangling — `corpo lint` will flag them for manual reconciliation.

# Technical Design

## CLI Interface

Three new commands handle the clone/pull/push lifecycle:

```
corpo clone <uri> [--title <title>] [--group <path>]
corpo pull <fileId>
corpo push <fileId>
```

For `corpo clone`, the user passes whatever URI is natural for the source — a local path, a GitHub URL, a full HTTPS URL. The CLI infers the adapter type:

| User input | Inferred adapter | Stored in `remote` |
|---|---|---|
| `./skills/frd/SKILL.md` or any relative/absolute path | `path:` | `path:skills/frd/SKILL.md` |
| `https://github.com/owner/repo/blob/main/file.md` | `git:` | `git:github.com/owner/repo/file.md` |
<!-- thread:9e1b5c83 -->

The `remote` value stored in frontmatter uses the normalized typed-prefix form regardless of what the user typed. `corpo pull` and `corpo push` read this normalized form to resolve the adapter.

## Adapter Interface

Each source type is an adapter implementing a minimal interface:

```typescript
interface RemoteAdapter {
  read(): Promise<string>            // Returns markdown body — required
  write?(content: string): Promise<void>  // Pushes body back to source — optional
}
```

Adapters are resolved from the URI prefix. The adapter registry is the extension point — adding a new source type means registering a new prefix and adapter class, with no changes to clone/pull/push command logic.

## Source URI Format

The `remote` frontmatter field uses typed URI prefixes:

| Prefix | Example | Pushable |
|---|---|---|
| `path:` | `path:skills/frd/SKILL.md` | Yes — direct file write |
| `git:` | `git:github.com/owner/repo/.corpo/files/{id}.md` | Yes — Contents API PUT |

Paths for `path:` are relative to the repo root. Absolute paths are not supported.

## Cut Complexity

`remote_files` in `.corpo/config.json` is deprecated. The key is ignored by current CLI versions with a migration warning. Existing entries are migrated by running `corpo clone` for each referenced URI — the result is a local corpus file with the same content and a stable ID that can be placed in navigation normally.

`~/.corpo/cache.db` is also retired. Previously, the cache stored fetched content for remote files so the GUI and CLI could serve them without a live network request. Under the snapshot model, the local corpus file is the working copy — there is nothing to cache separately. The user-level `~/.corpo` directory and any files reindexed there are no longer needed and can be removed after migration.

The semantic difference from the old model: `remote_files` fetched content live on every read. Cloned files are snapshots. Freshness is the author's responsibility. This tradeoff is intentional — live fetching introduces network dependencies into every CLI and GUI read path, and makes the corpus unreliable offline.

## Push Behavior

`corpo push <fileId>` reads the `remote` field to determine the target and resolves the adapter. What gets sent depends on the target type:

- **Non-corpo targets** (`path:`): body only, frontmatter stripped — the target format has no frontmatter concept.
- **Corpo targets** (`git:`): the corpus file is written back with local-only fields stripped (`remote`, `syncedAt`, `checksum`) — these belong only to the clone and must not pollute the source; `title`, `description`, `threads`, and body are written back so those changes propagate to the source repo.
<!-- thread:6f4a2d9e -->

The corpus file itself is not modified by push. Push is idempotent — running it twice with no intervening edits is a no-op at the destination.

## Auth

The `git:` adapter requires a GitHub token. Auth is resolved using the same credential store used elsewhere in the CLI — no additional setup beyond an existing GitHub token is needed.
