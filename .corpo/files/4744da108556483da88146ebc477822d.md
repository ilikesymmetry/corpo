# Overview

`corpo import <uri>` brings any external markdown-compatible content into the corpus as a first-class file. The result is a normal corpus file — stable ID, frontmatter, full threading — with the source URI stored in the `importSource` field for traceability. `corpo export <fileId>` pushes the edited body back to the original source for adapters that support writing. This pair replaces the `remote_files` config key, which referenced external content live without creating a local copy. Under the new model, all corpus content is local; the import/export boundary is explicit. As a side effect, the `~/.corpo/cache.db` entry for remote files becomes unnecessary — the local corpus file is the working copy, so no separate cache layer is needed.

## Goals

1. Any markdown-compatible external source can become a corpus file via `corpo import <uri>`
2. Edited corpus files can be pushed back to their source via `corpo export <fileId>` for adapters that support it
3. `remote_files` is retired and the `~/.corpo` remote content cache is eliminated — both are replaced by local corpus files

## Non-Goals

- Automatic or live sync — re-import is always an explicit author action
- Custom source adapters beyond local paths and GitHub files — Google Docs, Confluence, Notion, and similar are out of scope for v1
- Rich-format fidelity — non-markdown source formatting is not preserved; content is treated as plain markdown
- Conflict resolution when the corpus file and source have both changed since import


**V1 scope is two milestones:** same-repo local files (`path:`), then cross-repo GitHub files (`github:`). Custom rich-format sources (Google Docs, Confluence, Notion) are future work.

## Other Requirements

- The `importSource` frontmatter field stores the source URI and is the single back-reference for export and re-import
- `importSyncedAt` (ISO 8601 timestamp) records when the body was last pulled from the source — lets authors know how stale their copy is
- `importChecksum` (SHA-256 of the imported body) allows `corpo import` to detect whether the source has changed since last sync without fetching the full content, and lets `corpo export` skip a no-op write
- Importing a URI that already exists as a corpus file (matched by `importSource` field) updates the body rather than creating a duplicate
- All local frontmatter (title, description, threads) is preserved on re-import; only the body is overwritten
- When the source is itself a corpus file (e.g. a remote corpo file), its frontmatter is stripped on import — only the markdown body is taken. The local copy's frontmatter is always the author's own.
- Adapters declare whether they support export — `corpo export` on a read-only adapter fails with a clear error
- The adapter interface is defined as a first-class extension point so new source types can be added without modifying core CLI code
- The `remote_files` config key and `~/.corpo` remote content cache are both removed from the protocol — this feature is their complete replacement

# User Flows

## Importing a Local File

Author runs `corpo new --import ./skills/frd/SKILL.md --group "Skills"`. The CLI reads the file, infers the source type from the input (local path → `path:` adapter), and creates a corpus file with the content as body. Title is auto-extracted from the file's first H1 or falls back to the filename; author can override with `--title`. The assigned file ID is printed. `importSource`, `importSyncedAt`, and `importChecksum` are written to frontmatter automatically.
<!-- thread:99e93dce -->

From this point the file is a normal corpus file. Threads can be added, content can be edited. The source path is retained for export.

## Importing a Remote Corpo File (Cross-Repo Sharing)
<!-- thread:b1b9fb78 -->

Instead of adding an entry to `remote_files`, author runs `corpo new --import https://github.com/owner/repo/blob/main/.corpo/files/{id}.md`. The CLI recognizes the GitHub URL, fetches the file via the GitHub Contents API, and creates a local corpus file. The normalized `github:owner/repo/.corpo/files/{id}.md` URI is stored in `importSource`. The local copy can be edited and threaded independently of the source repo.
<!-- thread:bd7d69f8 -->

This replaces the previous cross-repo syncing mental model: rather than a live reference, you take a snapshot, work with it locally, and optionally push changes back.

## Importing a Google Doc

Author runs `corpo import gdoc:https://docs.google.com/document/d/{id}`. The CLI fetches the document via the Docs API and converts it to markdown. A corpus file is created with the content as body and the gdoc URI in `importSource`. Author can now thread, annotate, and hand the file ID to an engineering agent as context.

## Exporting Changes Back

Author has edited a corpus file originally imported from a local path. Runs `corpo export {fileId}`. The CLI writes the corpus file body (without frontmatter) back to the path stored in `importSource`. For API-based adapters, auth is required.

## Re-Syncing Import Source
<!-- thread:9078abc5 -->

Author knows the source document has been updated since import. Runs `corpo new --import ./skills/frd/SKILL.md` again (same URI). The CLI matches the existing corpus file by `importSource` field, overwrites the body with fresh content, and preserves all frontmatter. Thread anchors referencing shifted line numbers become dangling — `corpo lint` will flag them for manual reconciliation.

# Technical Design

## CLI Interface

`corpo import` does not exist as a separate command. Import is a flag on `corpo new`:

```
corpo new [--title <title>] [--description <desc>] [--import <uri>] [--group <path>]
```

When `--import <uri>` is provided, the CLI fetches content from the source to populate the body. The user passes whatever URI is natural for the source — a local path, a GitHub URL, a full HTTPS URL. The CLI infers the adapter type:

| User input | Inferred adapter | Stored in `importSource` |
|---|---|---|
| `./skills/frd/SKILL.md` or any relative/absolute path | `path:` | `path:skills/frd/SKILL.md` |
| `https://github.com/owner/repo/blob/main/file.md` | `github:` | `github:owner/repo/file.md` |
| `https://any-other-url` | `https:` | `https://any-other-url` |

The `importSource` value stored in frontmatter uses the normalized typed-prefix form regardless of what the user typed. `corpo export` reads this normalized form to resolve the adapter.

Re-syncing uses the same flag: `corpo new --import <uri>` on a URI that already has a matching `importSource` in the corpus updates the body rather than creating a new file.

## Adapter Interface

Each source type is an adapter implementing a minimal interface:

```typescript
interface ImportAdapter {
  read(): Promise<string>            // Returns markdown body — required
  write?(content: string): Promise<void>  // Pushes body back to source — optional
}
```

Adapters are resolved from the URI prefix. The adapter registry is the extension point — adding a new source type means registering a new prefix and adapter class, with no changes to import/export command logic.

## Source URI Format

The `importSource` frontmatter field uses typed URI prefixes:

| Prefix | Example | Exportable |
|---|---|---|
| `path:` | `path:skills/frd/SKILL.md` | Yes — direct file write |
| `gdoc:` | `gdoc:https://docs.google.com/document/d/{id}` | Yes — Docs API, lossy |
| `github:` | `github:owner/repo/.corpo/files/{id}.md` | Yes — Contents API PUT |
| `https:` | `https://example.com/doc.md` | No — arbitrary URL |

Paths for `path:` are relative to the repo root. Absolute paths are not supported.

## Replacing `remote_files`

`remote_files` in `.corpo/config.json` is deprecated. The key is ignored by current CLI versions with a migration warning. Existing entries are migrated by running `corpo import` for each referenced URI — the result is a local corpus file with the same content and a stable ID that can be placed in navigation normally.

The semantic difference from the old model: `remote_files` fetched content live on every read. Imported files are snapshots. Freshness is the author's responsibility. This tradeoff is intentional — live fetching introduces network dependencies into every CLI and GUI read path, and makes the corpus unreliable offline.

## Export Behavior

`corpo export <fileId>` reads the `importSource` field to determine the target, resolves the adapter, and calls `write(body)` with the corpus file body (frontmatter stripped). The corpus file itself is not modified. Export is idempotent — running it twice with no intervening edits is a no-op at the destination.

For `path:` and `github:` adapters, export is lossless. For `gdoc:`, the document is overwritten with the markdown content — rich formatting from the original document is not restored.

## Auth

API-based adapters (gdoc, github) require auth tokens. Auth is resolved per-adapter using the same credential store used elsewhere in the CLI. For `github:`, the existing GitHub token suffices. For `gdoc:`, a Google OAuth token with `https://www.googleapis.com/auth/documents` scope is required — the CLI initiates Device Flow if no token is cached.
