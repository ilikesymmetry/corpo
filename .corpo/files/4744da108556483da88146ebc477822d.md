# Overview

`corpo import <uri>` brings any external markdown-compatible content into the corpus as a first-class file. The result is a normal corpus file — stable ID, frontmatter, full threading — with the source URI stored in the `import` field for traceability. `corpo export <fileId>` pushes the edited body back to the original source for adapters that support writing. This pair replaces the `remote_files` config key, which referenced external content live without creating a local copy. Under the new model, all corpus content is local; the import/export boundary is explicit.

## Goals

1. Any markdown-compatible external source can become a corpus file via `corpo import <uri>`
2. Edited corpus files can be pushed back to their source via `corpo export <fileId>` for adapters that support it
3. `remote_files` is retired — cross-repo and remote content is imported rather than referenced live

## Non-Goals

- Automatic or live sync — re-import is always an explicit author action
- Rich-format fidelity — Google Docs formatting is not preserved; content is treated as markdown
- Conflict resolution when the corpus file and source have both changed since import

## Other Requirements

- The `import` frontmatter field stores the source URI and is the single back-reference for export and re-import
- Importing a URI that already exists as a corpus file (matched by `import` field) updates the body rather than creating a duplicate
- All frontmatter (title, description, threads) is preserved on re-import; only the body is overwritten
- Adapters declare whether they support export — `corpo export` on a read-only adapter fails with a clear error
- The adapter interface is defined as a first-class extension point so new source types can be added without modifying core CLI code

# User Flows

## Importing a Local File

Author runs `corpo import path:skills/frd/SKILL.md`. The CLI reads the file, creates a corpus file with the content as the body and `import: path:skills/frd/SKILL.md` in frontmatter. The assigned file ID is printed. Author places it in navigation with `corpo mv`.

From this point the file is a normal corpus file. Threads can be added, content can be edited. The source path is retained for export.

## Importing a Google Doc

Author runs `corpo import gdoc:https://docs.google.com/document/d/{id}`. The CLI fetches the document via the Docs API and converts it to markdown. A corpus file is created with the content as body and the gdoc URI in `import`. Author can now thread, annotate, and hand the file ID to an engineering agent as context.

## Importing a Remote Corpo File (Cross-Repo Sharing)

Instead of adding an entry to `remote_files`, author runs `corpo import github:owner/repo/.corpo/files/{id}.md`. The CLI fetches the file via the GitHub Contents API and creates a local corpus file. The source is tracked in `import`. The local copy can be edited and threaded independently of the source repo.

This replaces the previous cross-repo syncing mental model: rather than a live reference, you take a snapshot, work with it locally, and optionally push changes back.

## Exporting Changes Back

Author has edited a corpus file originally imported from a local path. Runs `corpo export {fileId}`. The CLI writes the corpus file body (without frontmatter) back to the path stored in `import`. For API-based adapters, auth is required.

## Re-Importing When the Source Has Changed

Author knows the source document has been updated since import. Runs `corpo import path:skills/frd/SKILL.md` again (same URI). The CLI matches the existing corpus file by `import` field, overwrites the body with fresh content, and preserves all frontmatter. Thread anchors referencing shifted line numbers become dangling — `corpo lint` will flag them for manual reconciliation.

# Technical Design

## Adapter Interface

Each source type is an adapter implementing a minimal interface:

```typescript
interface ImportAdapter {
  read(): Promise<string>            // Returns markdown body — required
  write?(content: string): Promise<void>  // Pushes body back to source — optional
}
```

Adapters are resolved from the URI prefix. The adapter registry is the extension point — adding a new source type means registering a new prefix and adapter class, with no changes to import/export command logic.

## Source URI Format

The `import` frontmatter field uses typed URI prefixes:

| Prefix | Example | Exportable |
|---|---|---|
| `path:` | `path:skills/frd/SKILL.md` | Yes — direct file write |
| `gdoc:` | `gdoc:https://docs.google.com/document/d/{id}` | Yes — Docs API, lossy |
| `github:` | `github:owner/repo/.corpo/files/{id}.md` | Yes — Contents API PUT |
| `https:` | `https://example.com/doc.md` | No — arbitrary URL |

Paths for `path:` are relative to the repo root. Absolute paths are not supported.

## Replacing `remote_files`

`remote_files` in `.corpo/config.json` is deprecated. The key is ignored by current CLI versions with a migration warning. Existing entries are migrated by running `corpo import` for each referenced URI — the result is a local corpus file with the same content and a stable ID that can be placed in navigation normally.

The semantic difference from the old model: `remote_files` fetched content live on every read. Imported files are snapshots. Freshness is the author's responsibility. This tradeoff is intentional — live fetching introduces network dependencies into every CLI and GUI read path, and makes the corpus unreliable offline.

## Export Behavior

`corpo export <fileId>` reads the `import` field to determine the target, resolves the adapter, and calls `write(body)` with the corpus file body (frontmatter stripped). The corpus file itself is not modified. Export is idempotent — running it twice with no intervening edits is a no-op at the destination.

For `path:` and `github:` adapters, export is lossless. For `gdoc:`, the document is overwritten with the markdown content — rich formatting from the original document is not restored.

## Auth

API-based adapters (gdoc, github) require auth tokens. Auth is resolved per-adapter using the same credential store used elsewhere in the CLI. For `github:`, the existing GitHub token suffices. For `gdoc:`, a Google OAuth token with `https://www.googleapis.com/auth/documents` scope is required — the CLI initiates Device Flow if no token is cached.
