# Corpo — PRD

Corpo is a git-backed, agent-native file protocol. Files are Markdown with
YAML frontmatter, stored in git repositories, and accessible to humans and
agents through persistent links. Designed for bottoms-up adoption — one author,
one repo, no mandate required.

## Architecture
<!-- thread:8f8f32e1 -->

### Files

Every Corpo file is a Markdown file with a YAML frontmatter block, named
`{file-id}.md`. The ID is a UUID v4 with hyphens stripped — 32 lowercase hex
characters. IDs are permanent; they never change even if the file moves.

Required frontmatter fields: `title`, `description`. Optional: `import` (set
when a file was created via `corpo import`, points to the original URL).

Threads live in frontmatter under a `threads` key. Each thread has an 8-char
hex ID, an author, timestamp, body, and optional replies. Thread position in
the body is marked with an HTML comment anchor: `<!-- thread:{id} -->`.

### Corpus

A corpus is a collection of files. Local files live in `.corpo/files/`. Remote
files are declared by ID in `.corpo/config.json` and fetched on demand.
`.corpo/config.json` governs the corpus with two keys:

- `remote_files` — map of `{id}: {remote}` for files fetched from elsewhere
- `navigation` — ordered group hierarchy covering local and remote files

### Cache and Sync

All fetched remote files are stored in a user-level SQLite cache at
`~/.corpo/cache.db`. The cache is binary — a file is either fully cached or
not present. Staleness is checked via blob SHA (git remotes) or API
(registry remotes). Reads are never blocked; background sync lands after.

**ID = identity. Remote = routing.** A file ID is a permanent global
reference. Where to fetch it from is a config/cache concern, never part of
the reference itself.

### Remotes

Two remote types:

- **Git remote** — `git:github.com/{owner}/{repo}.git` — fetches via the
  git host API using existing credentials
- **Registry remote** — `https://{host}` — HTTP service implementing the
  Corpo registry API (`GET /file/{id}`, `GET /file/{id}/content`)

Resolution order: cache hit → remotes in config order → error.

### Auth

Pluggable. GitHub OAuth ships first (repo visibility = file visibility).
Okta SSO and repo-level access tokens follow.

## Key Design Decisions
<!-- thread:c3d4e5f6 -->

- **Filename is the ID.** No `id` field in frontmatter — the filename is the
  sole source of truth for a file's identity.
- **Thread anchors are HTML comments.** `<!-- thread:{id} -->` is invisible
  to all Markdown renderers and unambiguous to parse.
- **Resolved threads live in git history only.** No sidecar archive file.
  corpo.sh backend may cache at resolution time for persistent link stability.
- **Corpus lives in `.corpo/`.** Config at `.corpo/config.json`, local files
  at `.corpo/files/`. Hidden by default; purpose-built clients are the
  intended interface.
- **No `local_files_root` in config.** Local files are always at
  `.corpo/files/` by convention. No configurable path.
- **Navigation is centralized in config.** Both local and remote files are
  placed in sidebar groups via `navigation` in `.corpo/config.json`. The first
  file in the first group is the implicit entry point for agents.
- **Cache is user-level, not project-level.** No per-corpus data store.
- **`corpo import` sets the `import` frontmatter field.** Enables dedup: CLI
  warns before creating a file with a duplicate `import` URL.
