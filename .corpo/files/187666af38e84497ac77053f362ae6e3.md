Resolves a thread via HTTP. Unlike the file commit endpoint, this endpoint
handles resolution logic server-side and creates a git commit automatically.

## Endpoint

### `DELETE /api/files/:fileId/threads/:threadId`

1. Reads `.corpo/files/{fileId}.md`
2. Validates the thread exists in frontmatter
3. Deletes `threads.{threadId}` from frontmatter
4. Removes `<!-- thread:{threadId} -->` (and trailing newline) from the body
5. Writes the updated file
6. Runs `git add .corpo/files/{fileId}.md`
7. Runs `git commit -m "thread resolved: {threadId} ({fileId})"`

**Response:** `204 No Content`
**Errors:** `404` file not found, `404` thread not found

## Commit Message Format

```
thread resolved: {8-char-threadId} ({32-char-fileId})
```

The format is intentionally specific so resolution commits are easy to find
in `git log` and independently revertable with `git revert`.

## Why Auto-Commit

Thread resolution is a semantic milestone — a conversation has been acted on
and closed. A discrete commit per resolution preserves history (who resolved
what and when) and makes each resolution independently revertable.

Other mutations (new threads, replies) don't auto-commit; they use the
author's normal git workflow. Resolution is the exception because it
permanently removes information from the file.
