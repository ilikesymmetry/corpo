# GUI Architecture

The GUI is a browser-based single-page app. The server is a thin I/O
layer — it reads and writes raw files, nothing more. All intelligence lives
in the client: frontmatter parsing, Markdown rendering, thread management,
and local state.

See [Server API](d8b4f3e2a1c54f9bb2e8c1d6f3e5b7a2) for the six endpoints.

---

## Client-Server Model

**Server** — exposes six endpoints against a storage backend. No parsing,
no rendering, no business logic. Runs locally as `corpo serve` or remotely
at corpo.sh.

**Client** — a browser SPA that owns all file intelligence:
- Parses raw YAML frontmatter and Markdown body
- Renders the navigation sidebar and file view
- Manages thread state in-memory (add, reply, resolve)
- Tracks unsaved changes and commits them as a full file write

The server API is the full contract. Same client codebase in all deployment
modes.

## Deployment Modes

### Local — `corpo serve`

```
corpo serve [--port 3000]
```

- Finds `.corpo/config.json` by walking up from CWD (same as git)
- Starts an HTTP server on `localhost` (default port 3000)
- **LocalAdapter**: reads from `.corpo/files/{id}.md`, writes on commit
- Client assets (JS, CSS) embedded in the binary — no CDN, no external deps
- Opens the browser automatically on start

### Hosted — corpo.sh

The same server runs at corpo.sh with a different storage adapter. The API
surface is identical. Self-hosted deploys follow the same pattern.

## Storage Adapter Interface

The server is backend-agnostic. Each method maps 1:1 to an API endpoint:

```
interface StorageAdapter {
  getNavigation()          → Navigation     // GET  /api/navigation
  putNavigation(nav)       → void           // PUT  /api/navigation
  getFile(id)              → string | null  // GET  /api/files/:id
  createFile(raw)          → string         // POST /api/files → returns id
  commitFile(id, raw)      → void           // PUT  /api/files/:id/commit
  deleteFile(id)           → void           // DELETE /api/files/:id
}
```

**LocalAdapter** — reads and writes the local filesystem:
- Navigation: `.corpo/config.json`
- Files: `.corpo/files/{id}.md`

**RegistryAdapter** — delegates to a remote corpo server. Used by corpo.sh
backends and by the CLI for remote file resolution.

## Client Model

### Startup

On load, the client fetches `GET /api/navigation` to get the navigation tree,
then renders the sidebar. File content is fetched lazily — `GET /api/files/:id`
— when the user navigates to a file.

### Parsing

The client parses each raw file directly:
- Splits on the frontmatter `---` delimiters to extract YAML and body
- Parses YAML to get `title`, `description`, `sidebarTitle`, `threads`
- Parses thread anchors (`<!-- thread:{id} -->`) from the body to position
  thread indicators inline

No server-side processing. The client is the parser.

### Local State

After fetching a file, the client holds its state in memory. All mutations
are applied to local state only:
- Content edits update the body
- Adding a thread updates `threads` in frontmatter and inserts an anchor in
  the body at the correct position
- Replying to a thread appends to `threads.{id}.replies`
- Resolving a thread removes the `threads.{id}` entry and the body anchor

The client tracks a dirty flag when local state diverges from the last
fetched version.

### Committing

When the user saves, the client serializes local state back to raw Markdown
(frontmatter + body) and posts it to `PUT /api/files/:id/commit`. The server writes
the file to storage. One write path for all mutation types.

```
fetch file → parse → mutate locally → serialize → commit
```

## URL Structure

```
/                           → redirect to first file in navigation
/{file-id}                  → render a specific file
/{file-id}#thread:{id}      → render file, scroll to thread anchor
```

URLs are stable — file IDs never change.

---

## GUI Rendering

**Sidebar** — built from the navigation tree returned by `GET /api/navigation`.
File labels are resolved from fetched file frontmatter:
`sidebarTitle` → `title`. Groups are collapsible.

**File header** — `title` and `description` displayed as a clean metadata
header. Raw YAML is never shown.

**Body** — Markdown rendered client-side: tables, syntax-highlighted code
blocks, heading anchor links. Thread anchors (`<!-- thread:{id} -->`) are
stripped from the rendered output and replaced with inline indicators that
open the thread panel.

**Design** — minimal, clean, purpose-built doc reader. Dark and light mode.
