# Server API

The corpo server exposes six endpoints. The API treats all files uniformly by
ID — whether a file lives on the local filesystem or is fetched from a remote
is an internal server concern. Clients never need to handle the distinction.

The same spec is implemented by `corpo serve` (local mode) and by the
corpo.sh backend (hosted mode). Clients are identical in both environments.

See [Viewer Architecture](c7a9b2e1f5d34c8a9b3e7f1d2c6b5a4e) for how this
fits into the broader design.

---

## Navigation

### `GET /api/navigation`

Returns the navigation tree. This is everything the client needs to render
the sidebar. File IDs are opaque to the client — local and remote files appear
identically.

**Response `200` — `application/json`:**
```json
{
  "navigation": [
    {
      "group": "Product",
      "children": [
        "5f0d0567687f4d628a8630e96ca20a83",
        { "group": "Discovery", "children": ["1c77eafa4db54e31baf4a22d4a03f2d8"] }
      ]
    }
  ]
}
```

Navigation is a recursive tree of nodes. Each node is either a file ID
(string) or a group object with a `group` label and a `children` array of
nodes. Files and subgroups can appear in any order within `children`.

---

### `PUT /api/navigation`

Replaces the navigation tree. Used by the client to reorder groups, rename
groups, add or remove file IDs from the sidebar, and add remote file IDs.
The server writes the updated navigation to config.json.

**Body — `application/json`:**
```json
{
  "navigation": [Group]
}
```

**Response:** `204 No Content`

**Errors:**
- `400` — missing or malformed `navigation`

## Files

All file endpoints are agnostic to file type. The server resolves a file ID
to its source (local filesystem or remote) internally.

### `GET /api/files/:id`

Returns the raw `.md` content for a file as `text/plain`. The client parses
frontmatter and body directly.

**Response `200` — `text/plain`:**
```
---
title: "Why we should change the fee model"
description: "..."
threads:
  3a7f2b9c:
    author: conner
    at: 2026-03-07T14:23Z
    body: "Does this hold at p99?"
---

Body content here.

<!-- thread:3a7f2b9c -->
```

**Errors:**
- `404` — file not found

---

### `POST /api/files`

Creates a new local file. The client provides the full initial raw content
including frontmatter. The server generates a UUID v4 (hyphens stripped) as
the file ID, writes the file, and returns the ID.

**Body — `application/json`:**
```json
{
  "raw": "---\ntitle: \"New file\"\ndescription: \"...\"\n---\n"
}
```

**Response `201` — `application/json`:**
```json
{ "id": "<file-id>" }
```

**Errors:**
- `400` — missing `raw`

---

### `PUT /api/files/:id/commit`

Writes the full raw file content to storage. The client sends the complete
`.md` — frontmatter and body — after applying all changes locally. This is
the single write path for all mutations: content edits, new threads, replies,
and thread resolution all produce a modified file committed via this endpoint.

**Body — `application/json`:**
```json
{
  "raw": "---\ntitle: \"...\"\n---\n\nBody here."
}
```

**Response:** `204 No Content`

**Errors:**
- `400` — missing `raw`
- `404` — file not found

---

### `DELETE /api/files/:id`

Removes a file from the corpus. The server handles all cleanup internally:
- **Local file** — deletes `.corpo/files/{id}.md` from disk
- **Remote file** — removes the entry from `remote_files` in config.json
- **Navigation** — removes the file ID from all groups in config.json

The file ID is fully purged in a single call. The client does not need to
follow up with `PUT /api/navigation`.

**Response:** `204 No Content`

**Errors:**
- `404` — file not found in local files or remote_files

---

## Errors

All errors return a JSON body:

```json
{ "error": "human-readable message" }
```

| Status | Meaning |
|--------|---------|
| `400` | Bad request — missing or invalid fields |
| `404` | File not found |
| `500` | Server error |

---

## Auth

In local mode (`corpo serve`), the server binds to `localhost` only. No auth
is applied — access is scoped to the local machine.

In hosted mode (corpo.sh), the auth scheme is backend-defined. The API
surface is otherwise identical.
