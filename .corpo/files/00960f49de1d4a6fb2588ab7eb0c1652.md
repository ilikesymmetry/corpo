# Corpo vs Obsidian

Obsidian is the closest prior art to Corpo in philosophy. Both are local-first,
plain-Markdown, git-compatible knowledge systems. The divergences are intentional
and driven by different problem domains: Obsidian is personal knowledge management
for a single user or small team sharing a vault; Corpo is an agent-native file
protocol for cross-team and cross-org document federation.

---

## Where They Align

| Dimension | Obsidian | Corpo |
|---|---|---|
| File format | Plain `.md` + YAML frontmatter | Plain `.md` + YAML frontmatter |
| Source of truth | Filesystem | Filesystem (git-backed) |
| Local-first | Yes — sync is optional | Yes — cache is local, reads never blocked |
| No binary formats | `.md` only | `.md` only |
| Sync is additive | Obsidian Sync is a paid add-on | Remote files fetched on demand |
| User-level config | Global app preferences | `~/.corpo/config.json` + `cache.db` |

Obsidian is an existence proof of the local-first, plain-file model at scale.
Corpo adopts this foundation wholesale.

---

## Key Divergences

### File identity: UUID vs human-readable filename

Obsidian uses human-readable filenames as primary identity (`[[Note Title]]`).
Renaming a file breaks links unless Obsidian is running to update them. Links
are fragile across external tools and across time.

Corpo uses UUID v4 filenames (hyphens stripped). IDs are permanent — renaming,
moving, or archiving a file never changes its ID. This enables stable persistent
links (`corpo.sh/{id}`) that survive any reorganization and are safely embedded
in code, external systems, and agent tool calls.

The tradeoff: UUID filenames are opaque without a client. Corpo accepts this
cost and invests in purpose-built CLI and viewer tooling as the human interface.
Raw filesystem browsing is not the target UX.

### Scope: single vault vs federated corpora

Obsidian is single-vault. Cross-vault linking has no native support — Obsidian
links resolve within one vault only. Teams sharing a vault share all files, all
plugins, all settings.

Corpo is designed for federation. Any corpus can declare remote files from
other git repositories or registry endpoints. A file's ID is globally unique
and can be referenced across org boundaries. A single Corpo link resolves
regardless of which corpus owns the file.

### Indexing: in-memory MetadataCache vs SQLite cache

Obsidian maintains an in-memory `MetadataCache` (links, tags, headings,
frontmatter) rebuilt on startup, persisted via IndexedDB (Electron's embedded
key-value store). The cache is local to the Obsidian app process.

Corpo uses a user-level SQLite cache at `~/.corpo/cache.db`. For local files,
the filesystem is the source of truth. For remote files, the cache is the
primary read path — content is written to `~/.corpo/files/{id}.md` on fetch
and served from there on subsequent reads.

### Navigation: filesystem hierarchy vs centralized config

Obsidian derives navigation from folder structure. Moving a file means moving
it on disk. No configuration required; any `.md` file in the vault is
automatically discovered.

Corpo centralizes navigation in `.corpo/config.json` under a `navigation` key. Groups
with ordered file ID lists define the sidebar structure. Local and remote files
are identical entries — just IDs. This approach is uniform across file
provenance (no different code paths for local vs remote) and allows remote
files to be placed in the viewer's navigation without inheriting the upstream
corpus's organizational scheme.

### Collaboration: no threading vs inline threads

Obsidian has no built-in threading or commenting. Collaboration requires
external tools or community plugins.

Corpo has a native thread primitive. Threads live in frontmatter under a
`threads` key; their position in the document is marked with an HTML comment
anchor (`<!-- thread:{id} -->`). The anchor is invisible to all Markdown
renderers. Resolved threads are removed from the file and preserved in git
history only.

### Agent participation: plugin API vs first-class protocol primitive

Obsidian exposes a plugin API (TypeScript, unsandboxed, full Node.js access in
Electron). Agents can interact with Obsidian via plugins, but the protocol was
not designed with agents in mind.

Corpo treats agents as first-class participants. `description` is a required
frontmatter field — a plain-text summary written for agents skimming the corpus.
The CLI supports field-level selection (`--fields title,description`,
`--no-threads`) so agents can read progressively. Navigation structure in
`.corpo/config.json` gives agents a structural index of the corpus in a single read.

## Where Corpo Goes Beyond Obsidian

| Capability | Obsidian | Corpo |
|---|---|---|
| Stable cross-org file references | No | Yes — UUID IDs, persistent links |
| Federation (remote corpora) | No | Yes — git and registry remotes |
| Agent-native progressive disclosure | No | Yes — `description` field, field selection |
| Inline collaborative threads | No | Yes — frontmatter threads + HTML anchors |
| Centralized navigation config | No | Yes — `navigation` in `.corpo/config.json` |
| `sidebarTitle` for display | No | Yes — optional frontmatter field |

---

## Design Decisions Informed by Obsidian

- **Plain `.md` + YAML frontmatter** — adopted directly. Obsidian proved this
  format works at scale for human and tool consumption.
- **Local-first cache** — Corpo's `~/.corpo/cache.db` is the direct analogue
  of Obsidian's IndexedDB: an embedded database for fast local reads without
  network calls. Separate file storage (`~/.corpo/files/`) mirrors how
  Obsidian keeps vault files as readable plain text.
- **Sync is additive** — remote files are fetched on demand; local files are
  always available. The network is never on the critical path for a read.
