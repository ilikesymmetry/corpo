# Session 6 — 2026-03-11

Clone/Pull/Push shipped end-to-end: CLI adapters, GitHub Pages GUI with auth and permissions, full test coverage, and follow-up fixes for push behavior and character encoding.

### Clone/Pull/Push CLI (`92ec78e`)

Implemented the CLI commands designed in the Session 5 FRD:

- `packages/cli/src/lib/remote.ts` — `RemoteAdapter` interface with two implementations: `PathAdapter` (local files, relative or absolute paths) and `GitAdapter` (GitHub Contents API, authenticated via `GITHUB_TOKEN`)
- `packages/cli/src/commands/clone.ts` — snapshots a `path:` or `git:` URI as a corpo file, writing `remote`, `syncedAt`, and `checksum` to frontmatter
- `packages/cli/src/commands/pull.ts` — re-syncs the body from the source URI, preserving all existing frontmatter
- `packages/cli/src/commands/push.ts` — writes the body (plus portable frontmatter for `git:` targets) back to the source; command logic owns field stripping, adapters own I/O
- `git:` URI normalization preserves branch/ref from the original URL
- 31 unit and integration tests across `clone.test.ts` and `remote.test.ts`, all passing

### GitHub Pages GUI (`92ec78e`)

Added a static hosting mode so the GUI can be deployed to GitHub Pages and read/write corpo files via the GitHub Contents API:

- `packages/gui/src/lib/adapter.ts` — `CorpoAdapter` interface with `LocalAdapter` (proxies to the local server) and `GitHubAdapter` (Contents API with SHA caching for writes; 409 conflicts surfaced to the user)
- `packages/gui/src/lib/auth.ts` — GitHub OAuth Device Flow with `localStorage` token persistence
- `packages/gui/src/lib/permissions.ts` — read/comment/write tiers with wildcard and domain email patterns; write defaults to `[]`
- `packages/gui/src/App.tsx` — switched wouter from history to hash routing for GitHub Pages compatibility; adapter selected at runtime based on `VITE_GITHUB_REPO`
- `packages/gui/vite.config.ts` — base path derived from `VITE_GITHUB_REPO` at build time
- `.github/workflows/deploy.yml` — GitHub Actions deploy workflow with required OIDC permissions
- `packages/cli/src/commands/init.ts` — now scaffolds `.github/workflows/deploy.yml` when run in a repo

### Thread anchor stripping on push (`60f6d59`)

`corpo push` now strips `<!-- thread:{id} -->` anchors from the body before writing to `path:` targets. Thread anchors are local metadata and should not leak into the source file being pushed to.

### README rewrite (`60f6d59`)

Rewrote `README.md` from a technical overview to a context-management layer framing, with separate install and workflow sections for agents and humans. Also cloned the README into the Local navigation group in `.corpo/config.json` so it is accessible via the GUI.

### UTF-8 decode fix (`a65750e`)

Fixed a silent data corruption bug in `packages/gui/src/lib/adapter.ts`: `atob()` returns a Latin-1 binary string, so multi-byte UTF-8 sequences (em dashes, curly quotes, etc.) were corrupted before YAML parsing. Replaced with `TextDecoder` for correct UTF-8 decoding of GitHub API base64 responses.

## Reflections

**Three core user journeys and where they're blocked.**

There are three flows that corpo is designed to bridge:

1. **Drafting** — Authors start from existing wikis and co-create new ideas with the agent, iterating through drafting and review rounds
2. **Sharing** — Drafted, pre-reviewed files are shared with teammates for further review and comment
3. **Cloning** — Reviewed files become reusable team context, cloned into any repo for efficient agent context insertion

**Drafting (1) is blocked on terminal/GUI friction.** The current split — write in terminal, review in GUI — adds enough friction to slow the iteration loop. The direction: bring ideation into the GUI (in-app chat for new file creation), add GUI buttons that trigger `/resolve-threads` and `/review` agent skills, and add a `/publish` skill + button that commits, pushes, and returns a Pages link.

**Sharing (2) is blocked on publish smoothness and commenting.** The Pages setup required a manual step (changing a repository setting) that should be automatable or avoidable. More critically, commenting and thread resolution were not working on the deployed Pages app — this needs to be debugged before sharing with teammates is viable. The direction: headless publish flow the agent can configure end-to-end, and fix the comment/resolve loop on the Pages frontend. Real editing stays local-only (agent runs locally as a technical constraint anyway), which actually enforces the intended agent-centric collaboration model.

**Cloning (3) is right directionally but needs reps.** The clone/pull/push primitives work, but the devx for someone cloning an entire wiki repo has not been tested. For new users, cloning a whole repo is conceptually simpler and operationally cheaper than per-file cloning — the hyper-sharding model may have been over-engineered for a scale that does not exist yet. The direction: figure out what the end-to-end new-user flow looks like and optimize for that.

These three journeys will drive the roadmap and the distribution strategy for the first launch to bottoms-up teams.
