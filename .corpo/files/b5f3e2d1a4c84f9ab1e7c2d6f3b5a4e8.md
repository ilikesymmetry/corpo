# Viewer Stack

The viewer is a Vite-built SPA embedded in the CLI binary. This file covers
the chosen stack, rationale for each decision, and notable alternatives.

See [Viewer Architecture](c7a9b2e1f5d34c8a9b3e7f1d2c6b5a4e) for how the
client fits into the broader system.

---

## Build Tool — Vite

Vite handles dev server (with HMR) and production bundling. Its output is
a static `dist/` directory that the CLI embeds at compile time and serves
at runtime — no user-facing install step for the viewer.

No strong alternative here. Vite is the standard for this kind of project.

---

## Framework — React

React with TypeScript. The user-facing client is entirely browser-based, so
framework weight affects initial load time but not install size. React's
bundle (~45KB gzipped) is acceptable for a local viewer.

**If bundle size becomes a constraint:** Preact is a drop-in swap — same
component API, ~3KB gzipped, enabled by a single Vite config alias. No code
changes required. Start with React; migrate to Preact later if needed.

Svelte and Solid were considered. Both have smaller runtimes, but React's
ecosystem is more relevant here (gray-matter, markdown-it, and other libs
have better React integration examples) and removes a learning curve.

---

## Styling — Tailwind CSS v4 + Typography Plugin

Tailwind v4 with the Vite plugin. Output is small (only used classes emitted)
and the utility-first approach is well-suited to the "minimal, clean doc
reader" aesthetic without introducing a component library dependency.

**`@tailwindcss/typography`** is required for styling markdown-it's HTML
output. Without it, rendered Markdown is unstyled — headings are the same
size as body text, paragraphs have no spacing. The plugin provides a `prose`
class that applies well-designed defaults to arbitrary HTML:

```html
<div class="prose dark:prose-invert max-w-none">
  <!-- markdown-it output -->
</div>
```

This handles heading hierarchy, paragraph spacing, list indentation, blockquote
styling, table borders, and inline code backgrounds. `prose-invert` handles
dark mode. This is the correct tool for this use case — it is designed
specifically for rendered Markdown content.

CSS Modules were considered for simplicity, but Tailwind's productivity
advantage is significant for building a polished UI from scratch.

## Markdown Rendering — markdown-it

**markdown-it** handles body rendering. It covers tables, fenced code blocks,
heading IDs for anchor links, and has a clean plugin API for custom rules.

Thread anchors (`<!-- thread:{id} -->`) are HTML comments and are stripped
from markdown-it output by default — they will not appear in the rendered
view unless explicitly handled. The client should scan for them in the raw
body before passing to the renderer and replace with placeholder elements.

**Syntax highlighting — highlight.js**

highlight.js with on-demand language loading. Core bundle is ~50KB; language
grammars are small and load lazily on first use. Integrates with markdown-it
via the `highlight` option.

Shiki (VS Code themes) was considered for aesthetics but rejected — it bundles
grammar files that push the total JS well past 1MB, too heavy for an embedded
viewer. highlight.js with a clean theme is sufficient.

---

## Frontmatter Parsing — gray-matter

**gray-matter** handles the `---` delimiter split and YAML parse in one call.
Returns `{ data, content }` — data is the parsed frontmatter object, content
is the raw Markdown body. Well-maintained, no problematic dependencies.

```ts
import matter from 'gray-matter'

const { data, content } = matter(raw)
// data.title, data.description, data.threads, etc.
// content → Markdown body (thread anchors still present)
```

---

## Routing — wouter

**wouter** (~2KB) handles the three viewer routes:

```
/                     → redirect to first file in navigation
/:id                  → render file
```

The `#thread:{id}` scroll target is handled client-side via `useEffect` after
render — not a distinct route.

React Router was considered and rejected as overkill for two patterns. A
fully custom router was also considered but wouter provides the same result
with less code.

---

## Asset Embedding

Vite builds to `dist/`. At CLI compile time, this directory is embedded
into the binary. At runtime, `corpo serve` serves the embedded assets
from memory.

The embedding mechanism depends on the CLI implementation language:

| Language | Mechanism |
|---|---|
| Go | `//go:embed dist` |
| Rust | `include_bytes!` or a build script |
| Node.js | Bundled into the package as static files |

The CLI language is not yet decided. Any of these approaches produces the
same result: a single binary that serves the viewer with no external
file dependencies.
