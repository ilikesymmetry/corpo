# Corpo vs Mintlify

Mintlify is a high-quality Markdown-based documentation platform. It informed
Corpo's navigation design significantly — particularly the centralized nav
config and `sidebarTitle` frontmatter field. The two systems differ sharply in
audience: Mintlify targets professional public-facing documentation sites; Corpo
targets internal team knowledge and agent-native file access.

---

## Navigation Architecture

### What Corpo adopted from Mintlify

**Centralized navigation config.** Mintlify declares all navigation in
`docs.json`. No file can inject itself into the sidebar — it must be listed
explicitly. Corpo adopts this model in `.corpo/config.json` under a `navigation` key.

The alternative (deriving navigation from `path` frontmatter, like Obsidian)
was considered but rejected: for remote files, `path` frontmatter reflects the
upstream corpus's organizational scheme, not the local corpus's. A remote
reorg would silently reorganize the local viewer's sidebar. Centralized nav
gives full control regardless of file provenance, and produces a single uniform
implementation path for the editor (no different code paths for local vs
remote files).

**Arbitrary-depth group nesting.** Mintlify supports recursive group nesting.
Corpo adopts the same — navigation is a recursive tree of nodes, where each
node is either a file ID (string) or a group with a `children` array of nodes.
Files and subgroups can be freely interleaved. Depth is unrestricted.

**`sidebarTitle` as a separate frontmatter field.** Mintlify separates sidebar
display name (`sidebarTitle`) from page title (`title`). Corpo adopts the same
split: `title` is the full document title; `sidebarTitle` is an optional short
label shown in the sidebar. Display label resolution: nav config explicit label
→ `sidebarTitle` → `title`.

### Where Corpo diverges

**Mintlify is fully centralized; Corpo nav covers local and remote files
identically.** In Mintlify, every page must be listed in `docs.json` or it
will not appear. In Corpo, every file is accessible by ID regardless of
navigation placement — files not listed in `navigation` are simply absent from
the sidebar but remain reachable via persistent link. No `hidden` field needed.

**Mintlify has tabs, anchors, products, versions, languages.** Mintlify's nav
schema is designed for multi-product public documentation sites with versioning
and i18n. Corpo's navigation schema is intentionally minimal: groups and files.
The additional Mintlify nav elements serve use cases (public docs, marketing
sites, multi-language) that are out of scope for Corpo's internal knowledge
protocol.

## File Format

| Dimension | Mintlify | Corpo |
|---|---|---|
| Format | `.mdx` (Markdown + JSX) | `.md` (plain Markdown) |
| Required frontmatter | `title` | `title`, `description` |
| Sidebar label | `sidebarTitle` | `sidebarTitle` |
| Component library | Cards, Steps, Tabs, Callouts, etc. | None (plain Markdown only) |
| Agent summary field | `description` (optional) | `description` (required) |

**On MDX.** Mintlify's JSX component library (Cards, Steps, Tabs, Callouts,
CodeGroup) meaningfully closes the gap with Google Docs for structured content.
Corpo deliberately deprioritizes MDX for Phase 1: the protocol commits to
plain `.md` for maximum portability and renderer compatibility. Rich components
are a viewer-layer concern deferred to a later phase.

**`description` is required in Corpo, optional in Mintlify.** Mintlify uses
`description` for SEO and its auto-generated `llms.txt`. Corpo requires it as
a first-class agent-readable summary — agents can progressively disclose a
corpus by reading only `title` and `description` from the cache without
fetching full file content.

## Agent-Native Design

Both systems have intentional agent-native investment, approached differently.

**Mintlify** auto-generates `llms.txt` and `llms-full.txt` from the nav
structure. Page `description` frontmatter feeds directly into `llms.txt`
entries. A right-click contextual menu offers "Open in Claude", "Open in
Cursor", "Add MCP" options. Agent support is layered on top of a human-first
documentation platform.

**Corpo** treats agents as first-class participants from the protocol level.
`description` is mandatory, not optional. The CLI supports field-level
selection (`--fields title,description`, `--no-threads`) so agents can read
progressively without fetching full file content. Navigation structure in
`.corpo/config.json` gives agents a structural index of the corpus in a single read —
equivalent to Mintlify's `llms.txt` but derived automatically from the same
config humans use for the sidebar.

## Audience and Use Case

| Dimension | Mintlify | Corpo |
|---|---|---|
| Primary audience | Developer-facing public docs | Internal teams + agents |
| Content type | Product documentation | Decisions, specs, context, threads |
| Access model | Public or auth-gated website | Git repo + registry, agent-accessible |
| Collaboration | None (read-only rendering) | Inline threads, git-backed writes |
| Navigation model | Centralized, explicit, editor-managed | Centralized in `.corpo/config.json` |
| File identity | Human-readable path | UUID (stable across renames) |
| Federation | None | Cross-repo remote file references |

Mintlify is optimized for publishing polished documentation to external readers.
Corpo is optimized for internal teams writing, discussing, and acting on files
alongside agents. A Mintlify site is a destination; a Corpo corpus is a
workspace.
