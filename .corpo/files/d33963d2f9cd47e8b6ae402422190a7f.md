title: "Drafting Experience"
description: "GUI improvements that close the loop between agent and author during file drafting: in-app agent action buttons (resolve-threads, review) and hot reloading of markdown content so the GUI stays live while an agent edits files."
---

# Overview

The drafting loop today splits across two surfaces: the agent chat window and
the GUI. Conversation and first draft happen in chat; reading, highlighting, and
commenting happen in the GUI. The friction point is returning to chat to kick
off the next agent action (a review pass, resolving threads). This feature
eliminates that round-trip by adding in-app agent action buttons to the GUI, and
closes the feedback gap by hot-reloading file content when the agent writes
changes to disk.

## Goals

1. Author never needs to leave the GUI to trigger a review or thread-resolution
   pass while in the same drafting session.
2. GUI content stays in sync with the filesystem within one second of an agent
   writing to a file — no manual refresh required.
3. Both improvements work with zero configuration in a standard `corpo serve`
   session.

## Non-Goals

- Replacing the initial drafting conversation in the agent chat window. Chat
  remains the right surface for open-ended conversation; the GUI supplements it.
- Async or queued agent actions. Actions fire against an active, watching agent
  session — there is no job queue, no persistence of pending requests.
- Support for arbitrary agent commands beyond `/review` and `/resolve-threads`.
  Those are the two high-value actions; all others remain in chat.
- Hot reload for navigation changes (sidebar) or file metadata changes (title,
  description in the header). This iteration targets body content only.

## Other Requirements

- Hot reload must not cause scroll position loss or re-render flash when content
  is unchanged.
- The action buttons are only rendered in local-server mode. GitHub Pages mode
  has no agent to dispatch to and no filesystem to watch.
- Triggering an action while a previous action is still running must be safe
  (debounced or disabled state while active).

# User Flows

## Triggering a Review Pass

Author has finished a first draft and moved to the GUI to read it through. They
want a design review from the agent without touching the chat window.

In the file view header or a persistent toolbar, the author sees a "Review" button.
They click it. The GUI posts an agent action signal for the current file ID. The
agent session — already open and watching — picks up the signal and runs the
`/review` skill against that file. Thread annotations begin appearing in the
thread panel as the agent writes them. The author sees threads populate in
real time (powered by hot reload) and reads feedback without ever switching
windows.

## Triggering Thread Resolution

Author has left comments on a draft (or the agent has left review threads) and
wants the agent to address them. They click "Resolve Threads" in the GUI. The
GUI signals the agent, which runs `/resolve-threads` on the current file. As
the agent rewrites sections and removes thread anchors, the file view updates
live. The thread panel empties as each thread is resolved. When the agent
finishes, the author sees a clean document with all threads gone.

## Hot Reload While Agent Edits

Agent is mid-session editing a file the author has open in the GUI. The author
is passively reading while the agent works. Whenever the agent writes a new
version of the file, the GUI detects the change via the SSE stream and
re-fetches the content. The view updates smoothly — the body re-renders with new
content, scroll position is preserved where possible. No refresh banner, no
stale content.

## No Active Agent Session

Author opens the GUI standalone (no Claude Code session running) and clicks
"Review." The button still appears, but after posting the action signal with no
response within a timeout, the GUI degrades gracefully: a tooltip or inline
message explains that an active agent session is required, and displays the
command to run (`/review`) that the author can paste into chat.

# Technical Design

## Agent Action Signaling

The GUI needs a channel to reach a running Claude Code session. The two
candidates are:

1. **Agentation annotations** — GUI posts an annotation of type `agent-action`
   via the Agentation MCP service. Claude Code, running `agentation_watch_annotations`,
   picks it up and dispatches the skill.
2. **Server-side signal file** — GUI calls `POST /api/agent/action` on the
   local server, which writes a signal to `.corpo/.pending-action`. Claude Code
   watches that path and consumes it.

Agentation is the preferred approach because the Agentation MCP server is already
integrated in this project and provides a clean bidirectional channel. The action
annotation payload:

```json
{ "type": "agent-action", "action": "review" | "resolve-threads", "fileId": "<id>" }
```

The GUI marks the button as "pending" while the action is in flight. Since there
is no explicit completion signal from the agent, the button returns to idle state
after the file's content next changes (hot reload event) or after a 60-second
timeout.

## SSE File Watch

The local server adds a new endpoint:

```
GET /api/watch
```

This endpoint holds an SSE connection open and emits events when files under
`.corpo/files/` change. The server uses `fs.watch()` (recursive) on the files
directory. On each change event, it emits:

```
event: file-changed
data: { "id": "<fileId>" }
```

The `useFile` hook subscribes to `/api/watch` on mount. When a `file-changed`
event arrives matching the currently-viewed file ID, the hook re-fetches via
`GET /api/files/:id` and re-parses. The existing `parseFile()` pipeline handles
the rest. If the SSE connection drops, the hook attempts reconnection with
exponential backoff (1s, 2s, 4s, cap 30s).

## Scroll Position Preservation

On a hot-reload re-render, `useFile` records the window's scroll offset before
updating state and restores it after the DOM settles (via `requestAnimationFrame`
after state flush). If the document grew shorter than the previous scroll
position, the browser natural scroll clamping handles it.

## UI Placement

The action buttons ("Review", "Resolve Threads") live in the file view header
row, right-aligned, adjacent to any existing controls. They are icon+label
buttons at small size, visually muted until hovered. They are only rendered
when the GUI's adapter is `LocalAdapter` (local server mode). In GitHub Pages
mode, the buttons are absent.
