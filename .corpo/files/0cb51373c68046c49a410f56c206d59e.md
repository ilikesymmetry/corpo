# Session Journal — 2026-03-08

First full implementation session. Went from zero code to a working local GUI
with commenting. Notes on what we built and what we learned.

## What We Built

### File housekeeping

Migrated the config from `config.yaml` to `config.json`. Updated all
navigation references, files, and the protocol spec. Changed the navigation
schema from `{ files, groups }` to a recursive `Node = string | { group, children }`
type — simpler, more flexible, allows free interleaving of files and subgroups.

Added five new Architecture files:
- **GUI Architecture** — client-server model, StorageAdapter interface, deployment modes
- **Server API** — the six-endpoint contract
- **API Server Implementation** — LocalAdapter class, Hono routes, navigation cleanup
- **GUI Stack** — Vite + React + Tailwind CSS v4 + markdown-it + wouter rationale
- **CLI Stack** — incur + Bun + Hono, real command implementations

Pruned the task list down to four active tasks (serve, GUI renderer, threads,
edit). Removed everything outside current scope. Tasks are now a temporary pad for
autonomous delegation, not a permanent tracker.

### CLI + API server

Built `corpo serve` using **incur** (TypeScript CLI framework by wevm) + **Hono** +
**Bun**. The server is thin — all logic lives in `LocalAdapter` which implements
six methods mapping 1:1 to the six API endpoints:

- `GET /api/navigation` — reads `.corpo/config.json`
- `PUT /api/navigation` — writes back
- `GET /api/files/:id` — reads `.corpo/files/{id}.md`
- `POST /api/files` — generates UUID, writes new file
- `PUT /api/files/:id/commit` — overwrites file
- `DELETE /api/files/:id` — deletes file + strips ID from navigation atomically

The server also serves the embedded GUI SPA from `gui/dist/` for all other
routes. In dev, the GUI's Vite server proxies `/api/*` to `localhost:3000`.

### GUI SPA

Built the GUI as a Vite + React + TypeScript SPA:

- **gray-matter** was the original frontmatter parser but uses Node's `Buffer` —
  doesn't work in browsers. Replaced with `js-yaml` + a simple regex delimiter split.
- **Sidebar** — recursive navigation tree from `GET /api/navigation`. File titles
  fetched lazily in parallel after navigation loads; IDs shown as fallback.
- **FileView** — title + description header, markdown body rendered via **markdown-it**
  with **highlight.js** syntax highlighting, **@tailwindcss/typography** prose styling.


### Threading

Highlight any text in a file body → a floating input appears below the selection.
Type a comment, press Enter to save. Shift+Enter for newlines, Escape to dismiss.

On submit:
1. Generates an 8-char hex thread ID
2. Inserts `<!-- thread:{id} -->` anchor into the raw body at the correct position
3. Adds thread entry to `meta.threads` in frontmatter
4. Serializes and commits via `PUT /api/files/:id/commit`

Threads render in a right panel alongside the content.

**Anchor placement bug + fix:** The first pass inserted anchors via text search
(`content.indexOf(selectedText)`). This fails for tables and other formatted
content because rendered DOM text doesn't match raw markdown. Fixed by adding a
markdown-it core rule that stamps `data-line-end` on every rendered block element.
At selection time, we walk up the DOM from the cursor position to read that attribute,
then insert by line number instead of text matching. Unit tested in
`gui/src/lib/serialize.test.ts`.

### `corpo reply` command

Added a `corpo reply <fileId> <threadId> <body>` command specifically for agent use.
Reads and writes the file directly on disk — no server required. Much cheaper to
invoke than a custom bash script.

```sh
bun /path/to/corpo/src/index.ts reply <fileId> <threadId> "Reply body here"
```

## What We Learned

- **incur** works exactly as documented. The `Cli.create().command().serve()` pattern
  is clean and gives `--llms`, `--mcp`, and `--schema` flags for free.
- **gray-matter is not browser-compatible.** js-yaml is. Worth remembering for
  any future library choices: check for Node built-in usage before adding to the
  GUI bundle.
- **markdown-it source maps** are the right primitive for mapping DOM selections
  back to source positions. `token.map[1]` (exclusive end line) is the insertion point.
- **Corpo files as a living document** — we added comments to the PRD during the session
  and they round-tripped correctly through the GUI, the API, and git. The protocol
  is working as intended.

## Session Continuation — 2026-03-09

Second block of work in the same session. Focused on GUI polish: layout controls,
thread UX, and bidirectional scroll sync.

### GUI layout controls

- **Sidebar collapse** — hover the sidebar to reveal a button on its right edge.
  Click to collapse to zero width (smooth CSS transition). Button flips direction
  to expand. State lives in `App.tsx`.
- **ThreadPanel collapse** — same pattern mirrored on the right side. State is
  self-contained in `ThreadPanel`.
- **Sidebar logo removed** — stripped the "corpo" wordmark from the sidebar top,
  leaving only navigation.
- **Chevron size** — group toggle chevrons (▾/▸) wrapped in `text-base` span so
  they render larger than the surrounding `text-xs` label text.

### Threaded heading highlights

Headings with an attached thread anchor now render with a subtle amber background
tint. Hover intensifies it. Clicking activates the thread.

**Bug fix:** The first pass matched headings by `textContent`, which broke because
`typographer: true` in markdown-it converts straight apostrophes to curly quotes in
the rendered DOM. Fixed by switching to line-number matching: `parseHeadingThreads`
returns `Record<lineNumber, threadId[]>` keyed on the 0-indexed line in the raw
content. The rendered `<h2>` already carries a `data-line` attribute (stamped by
the existing `source_map` core rule), so the lookup is exact.

### Bidirectional scroll sync

Clicking a threaded heading → activates the thread → ThreadPanel smoothly scrolls
to center the thread card. Clicking a thread card → activates the thread → article
smoothly scrolls to center the heading.

Scroll is strictly unidirectional: only the *opposite* side scrolls on activation.
Implemented with two separate tick counters (`panelScrollTick`, `articleScrollTick`)
so `useEffect` dependencies are independent. This avoids the jarring behavior of
the clicked side scrolling while the user is reading it.

### Reply input

Active threads now show a single-line textarea at the bottom of the thread card.
`Enter` submits the reply; `Shift+Enter` adds a newline. On submit, the reply is
appended to `localMeta.threads[id].replies` in FileView and committed via
`PUT /api/files/:id/commit`. The input only appears when a thread is selected —
not on hover — to avoid jitter from height changes during mouse movement.

## Session Continuation — 2026-03-09 (continued)

Third block of work in the same session. Thread resolution, documentation
cleanup, and `corpo new`.

### Thread resolution

Human reviewers can now resolve threads directly from the GUI. Hover any
thread card to reveal a 14px SVG checkmark in the top-right corner. Clicking it:

1. Optimistically removes the thread from local state (no flicker)
2. Calls `DELETE /api/files/:fileId/threads/:threadId`
3. Server removes the `threads.{id}` frontmatter entry and `<!-- thread:{id} -->`
   body anchor, then auto-commits the file with message
   `thread resolved: {threadId} ({fileId})`

The checkmark is `absolute top-2 right-2` inside a `relative group/card`
container — zero layout shift whether visible or hidden.

Also added `corpo resolve <fileId> <threadId>` as a CLI command (plain file
write, no auto-commit — consistent with `corpo reply`). The API endpoint is
the only path that auto-commits, because resolution is a semantic milestone
worth a discrete commit.

### Thread anchor integrity lint + PostToolUse hook

Added `corpo lint [fileId]` to validate 1:1 correspondence between frontmatter
threads and body anchors. Two rules: `orphaned-thread` and `dangling-anchor`.
Code fences are stripped before scanning to avoid false positives from
documentation examples.

Registered as a Claude Code `PostToolUse` hook in `.claude/settings.json`:
fires automatically after every Edit or Write on a `.corpo/files/*.md` file.
Errors surface in the conversation immediately, before any git commit.

Also added `corpo threads <fileId>` — lists all threads as JSON with a
`needsResponse` flag (true when the last author isn't `claude`). Used by the
`resolve-threads` skill to find actionable threads.

### `corpo new` and shared createCorpoFile()

Added `corpo new <title> <description>` to scaffold a new corpo file without
reaching for Python. Outputs `{ id, path }`.

Both `corpo new` and `POST /api/files` now call `createCorpoFile(root, raw)`
from `src/lib/files.ts`. That function owns the ID generation + write logic;
neither callsite duplicates it.

### Features group

Added a `Features` subgroup under `Product` in the navigation, with three
nested subgroups: `CLI`, `API`, `Client`. 14 new feature files documenting
every implemented feature at each layer.

Updated `CLAUDE.md`: removed the stale "CLI doesn't exist" note, corrected
phase statuses, and added a **Sync Principle** — corpo files must
stay in sync with implementation; stale docs are a bug.

## Session Continuation — 2026-03-09 (continued again)

Fourth block. Focused on distribution, DX polish, and agent-friendly CLI primitives.

### Published to npm as @ilikesymmetry/corpo 0.1.0

Attempted to publish as `corpo` — blocked by npm similarity policy (too close
to `cors` / `copy`). Switched to scoped package `@ilikesymmetry/corpo` with
`--access=public`. `prepublishOnly` runs `build:gui` automatically so the GUI
dist is always bundled. Updated README install command accordingly.

### viewer → gui rename

Renamed `packages/viewer/` to `packages/gui/` across the entire codebase:
directory, package name, build scripts, `.gitignore`, `.npmignore`, path refs
in the server, and all ~18 corpo feature files. "Viewer" was generic;
"GUI" maps to the 3-interface framing (CLI / API / GUI).

### Bun install onboarding

Two touchpoints improved:
- **README**: Bun install is now step 1 with a runnable `curl` command before
  the npm install step.
- **`bin/corpo.js` error**: When bun is not found, the shim now prints both
  `curl` and `npm install -g bun` as runnable alternatives (npm is always
  available since Node is guaranteed present in an npm install context).

### Corpo skill self-documentation

Updated `skills/corpo/SKILL.md` to be more self-contained for agents:
added package identity (`@ilikesymmetry/corpo`, Bun ≥ 1.0), Step 1 CLI
availability check with full install instructions, source-repo invocation path,
`--help` progressive disclosure note, all 7 commands in the table, and a
lint-after-edit reminder.

### Navigation placement: corpo new --group and corpo mv

`corpo new` now accepts an optional `--group <path>` flag to atomically create
a file and insert it into the navigation tree in one command. `corpo mv <id>
<path>` is a new command to move an existing file to a different navigation
group. Both use `/`-delimited group labels. Missing groups are created
automatically. Moving to root uses an empty string.

Four pure nav helpers added to `src/lib/config.ts`:
`navInsert`, `navRemove`, `navContains`, `parseGroupPath`.

### corpo update

New command. Fetches `registry.npmjs.org` for the latest version, compares
with the version embedded in `package.json`, and runs `npm install -g
@ilikesymmetry/corpo@latest` if outdated. No-ops with a message when current.

### corpo serve --detach

Added `--detach` flag to `corpo serve` for non-blocking agent use. When set,
spawns a background server process via `Bun.spawn` with `detached: true` using
`process.execPath` + `process.argv[1]` (works in both source and installed
contexts), unrefs the child, and returns `{ url, pid, detached: true }`
immediately. Default blocking behavior unchanged for human terminal use.

The skill now instructs agents to always pass `--detach` when invoking
`corpo serve` programmatically.

### Dev scripts: bun run dev and bun run dev:bg

Added two convenience scripts to the root `package.json`:
- `dev`: runs `dev:cli & dev:gui & wait` — both servers in one terminal,
  Ctrl+C kills both. Human use.
- `dev:bg`: uses `corpo serve --detach` for the API + `nohup ... &` for Vite,
  returns immediately with PIDs. Agent use (avoids blocking the Bash tool).
  Vite logs go to `/tmp/corpo-gui.log`.

## What's Next

The general mode from here is iteration-driven: use the tool, notice friction,
communicate it, smooth it out. No big project plan required. A few directions
worth focusing on:

### 1 — Multi-repo

The killer feature is using corpo across multiple repositories — fetching files
from a remote git repo into a local session, building shared context across
projects. This is the core of the original protocol vision and the most
important thing to wire up next.

### 2 — Operations cleanup

Make the CLI a proper executable (not `bun src/index.ts`). Add a real test
harness and CI. General hygiene to make future iteration safe and fast,
and to limit accumulating tech debt before it compounds.

### 3 — Human↔agent handoff refinement

The Agentation model — annotating a live UI, having the agent pick up
annotations automatically — points toward something more general: tighter,
faster feedback loops between human and agent when refining the knowledge base.
Worth thinking about how to bring that same pattern into the corpo workflow
itself.
