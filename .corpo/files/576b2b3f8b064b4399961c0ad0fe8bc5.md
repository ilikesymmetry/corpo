title: "File API"
description: "Four endpoints for corpus file lifecycle: fetch (GET), create (POST), commit (PUT), and delete (DELETE). All file IDs are treated uniformly — the server resolves local vs. remote internally."
---

Four endpoints cover the full file lifecycle.

## `GET /api/files/:id`

Returns raw `.md` content as `text/plain`. The client parses frontmatter and
body directly — no server-side processing.

**Response `200`:** raw markdown text
**Errors:** `404` file not found

## `POST /api/files`

Creates a new local file. The server generates a UUID v4 (hyphens stripped)
as the file ID, writes to `.corpo/files/{id}.md`, and returns the ID.

**Body:** `{ "raw": "---\ntitle: ...\n---\n\n..." }`
**Response `201`:** `{ "id": "<file-id>" }`
**Errors:** `400` missing `raw`

## `PUT /api/files/:id/commit`

Overwrites a file's full content. The client sends the complete serialized
file — frontmatter and body — after applying all changes locally. This is the
single write path for all mutation types: content edits, new threads, replies.

**Body:** `{ "raw": "..." }`
**Response:** `204 No Content`
**Errors:** `400` missing `raw`, `404` file not found

## `DELETE /api/files/:id`

Removes a file from the corpus and all navigation references atomically:
- Deletes `.corpo/files/{id}.md` from disk
- Removes the ID from all groups in `.corpo/config.json`

**Response:** `204 No Content`
**Errors:** `404` file not found
